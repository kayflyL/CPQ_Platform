# 设计原则

> 版本：v0.1.11 | 更新日期：2026-07-11

① 渐进式重构

严禁推倒重写。每个版本必须是可运行的，每一步修改必须可回滚。

--------------------------------

② 系统稳定性优先

任何新功能或重构，都不能导致现有核心功能（报价计算、导出）失效。

--------------------------------

③ 修改文件必确认

在调用 `write_file`、`patch`、`terminal (修改操作)` 之前，**必须**向用户说明修改意图、影响范围，并获得明确的"确认"或"执行"指令，绝不静默覆盖用户未授权的配置文件。

--------------------------------

④ 硬编码零容忍

所有文件读写必须基于 `Path(__file__).parent` 动态获取，确保系统移动到任何盘符/电脑均可运行。

使用 `venv` 或 `requirements.txt` 管理依赖，绝不依赖用户电脑的全局环境。

--------------------------------

⑤ DB 变更保护

涉及数据库结构变更（`ALTER TABLE`）或批量删除数据前，必须先执行物理备份。
删除业务数据（如项目）必须进入"回收站"，禁止直接物理抹除，除非用户明确要求。

--------------------------------

⑥ 敏感信息保护

`opportunities.db`、`kp_data.db` 等核心数据库在 Git 提交时必须被 `.gitignore` 忽略。

--------------------------------

⑦ 验证后交付

如果网络/工具报错导致无法验证，必须如实汇报阻塞点，绝不伪造成功日志。

--------------------------------

⑧ 项目目录整洁

项目根目录保持整洁清晰，避免散落文件。所有文档（CHANGELOG、架构说明、流程图等）统一放在 `docs/` 文件夹。新增文件时优先考虑目录结构的可维护性，确保项目易于导航。

--------------------------------

⑨ 公共组件优先抽取

当同一 UI 模块被多个页面引用时，必须抽取为独立公共组件，禁止复制粘贴代码。若有 2 个以上页面需要相同功能，即为抽取阈值。

--------------------------------
