# 服务器页改造 · 落地设计文档

> 本文档是原型（`D:\CPQ_Platform_V1\server-config-prototype.html`）验证通过后的落地执行蓝图。
> **改造授权**：大刀阔斧按新设计重写到位，不保留旧表过渡、不搞新旧并行、不写兼容层（用户明确授权）。

---

## 1. 背景与目标

**痛点**：服务器报价 = L6 机箱价 + KP 配件价；L6 多系列（Orion=AMD / Polaris=兆芯）理不顺；服务器页与报价工作台职责重叠；KP 选配与机箱周边组件无联动，全靠人脑。

**目标**：服务器页改造为「配置 + 管理」双模式——配置面走 选机型 → 四步配置（基准/前面板/后面板/PSU）+ KP 选配，核心配件自动推导机箱周边；管理面统一管料号库 + 用户自己挑件组建基准配置。工作台复用同一套配置流程，出口接报价。

**关键原则**：一切料号收敛进统一料号主表；推导规则可配置；L6 与 KP 同构（都是"从料号库选件组 BOM"）。

---

## 2. 总体架构

```
服务器页 /servers（L6Pricing.vue 重写）
├─ 配置模式（消费侧）
│   选服务器类型 → 机型目录 → 四步配置 + KP 选配 → 生成 BOM（服务器页出无价 BOM，可分享）
│   └─ 四步：① 基准配置(摘要+详情) ② 前面板线缆(按硬盘推导) ③ 后面板IO槽位(选配) ④ 电源(功耗推导)
│   └─ 推导引擎：背板类型/GPU架构/电源/线缆/GPU供电线 由选配推导，可手改、留痕
└─ 管理模式（维护侧）
    ├─ 料号库：所有料号（L6 + KP）逐条 CRUD，按类别浏览
    └─ 基准配置组装：从料号库挑公共底盘件 + 选背板 → 存成基准方案（带盘位匹配校验）

报价工作台 /workspace（联动）
└─ 中栏 L6ConfigCard 替换为"配置流程核心"（与服务器页配置面共用组件），出口接报价逻辑（算价/利润/汇率/生成报价单）
```

服务器页配的 = 配置方案/无价 BOM（不绑商机、可分享）；工作台配的 = 报价配置（绑商机、含价）。两者配置明细同构、共用流程，分两张主表存。

## 3. 数据模型（PostgreSQL，l6 / kp / rules schema）

> **废弃**：`l6_base_config_parts`（旧冗余）、`l6_front_panel_items`、`l6_rear_panel_items`、`l6_psu_options` 的料号字段、预留的 `l6_bom_templates`/`l6_bom_parts`。料号统一进 `parts_master`。

| applicable | JSONB | 适用性：{form:[], bays:[], backplane:[], series:[]}（按机型收敛用，继承现有 applicable_* 思路） |
| sort_order | INT | |

## 4. 数据迁移（一次性，旧表迁移后废弃）

迁移脚本 `scripts/migrate_to_parts_master.py`：

1. 建 §3 全部新表。
2. **料号归并**（去重 by pn）：
   - `l6_base_config_parts` 的 pn → parts_master(category 按件推断)
   - `l6_front_panel_items` → parts_master(category=前面板线缆, specs={kind:cable_type, group_size, applicable_bays, applicable_backplane})
   - `l6_rear_panel_items` → parts_master(category=后面板Riser/模组/OCP, specs={io_slot, option_type, applicable})
   - `l6_psu_options` → parts_master(category=电源, specs={wattage, applicable_chassis})
   - `kp_parts` → parts_master(category=CPU/GPU/内存/硬盘/..., specs={tdp, kind, cables_per, ...})
3. **基准配置关系迁移**：旧 base_config_parts（config_id+pn+qty）→ 新 base_config_parts（config_id+pn+qty+locked=true）；旧 base_configs 补 bp_tri_pn/bp_dc_pn（按 backplane_type 推断）。
4. 建 server_types / server_models（从现有 base_configs 的维度反推几条机型记录作种子）。
5. 推导规则种子数据（§6 的默认值写入 derivation_rules）。
6. **废弃旧表**：DROP l6_front_panel_items / l6_rear_panel_items / l6_psu_options / l6_bom_templates / l6_bom_parts；旧 l6_base_config_parts 重命名备份后由新版替代。

> 迁移前 pg_dump 全库备份；迁移脚本 dry-run 校验料号无丢失、基准配置清单条数一致。

## 5. 后端设计

### 5.1 Repository 层（按新设计重组，废弃 L6ChassisRepository 旧 4 套平行方法）
- `PartsMasterRepository`：料号 CRUD（list 按类别/筛选、get_by_pn、insert、update、delete），读 specs/applicable JSONB。
- `BaseConfigRepository`：基准配置 CRUD + 底盘件清单（引用 pn，JOIN parts_master 取 name/price）；含盘位校验（bp_tri_pn/bp_dc_pn 的 specs.bays == config.bays）。
- `ServerCatalogRepository`：server_types / server_models CRUD。
- `ConfigSchemeRepository`：保存/读取配置方案（无价 BOM）。
- 复用现有 `kp_repo`（价格历史等业务）。

### 5.2 推导引擎 `engine/derivation_engine.py`（纯算法，不碰 DB，接收状态返回推导结果）
输入：`{base_config, kp_lines[], gpu_arch, rear_slots, overrides}` → 输出各步推导值 + 依据 + 校验警告。
- `derive_bp_type(kp_lines)`：含 SATA/SAS → tri，纯 NVMe/无盘 → dc
- `derive_power(kp_lines)`：500 + ΣCPU tdp + ΣGPU tdp；qty = power>4800?4:2；wattage = min(≥power/路数)
- `derive_front_cables(kp_lines)`：每种硬盘 ÷ group_size（读 parts_master.specs.group_size）
- `derive_gpu_cables(kp_lines)`：Σ 卡数 × specs.cables_per（按显卡型号）
- `switch_extra(gpu_arch)`：switch → NVSwitch 板 + NVLink
- `check_constraints()`：Switch 必 8 卡；背板盘位匹配；返回 warnings[]
> 规则参数从 `derivation_rules` 表读（可配置），引擎只执行；功耗阈值/分组数/单卡线数都查表，不写死。

### 5.3 API 路由
- `api/parts.py`：GET /api/parts?category=、POST/PUT/DELETE /api/parts/{pn}
- `api/base_configs.py`：GET /api/base-configs、GET /api/base-configs/{id}（带底盘件清单）、POST/PUT/DELETE、POST /api/base-configs/{id}/parts（组装/改清单）
- `api/server_catalog.py`：GET /api/server-types、GET /api/server-models?type=
- `api/derive.py`：POST /api/derive（传当前配置状态，返回推导结果 + 校验）
- `api/config_schemes.py`：POST /api/config-schemes（保存无价 BOM）、GET 列表/详情
- 工作台报价相关接现有 `api/quote.py`，新增保存 quote_configs。

## 6. 推导规则清单（固化进 derivation_rules，默认值）

| rule_key | 默认 params | 说明 |
|---|---|---|
| base_power | {watts:500} | 机箱基础功耗 |
| psu_qty_threshold | {watts:4800, qty_low:2, qty_high:4} | 电源数量：>阈值→4(2+2)，≤→2(1+1) |
| psu_redundancy | {mode:"1+1_low,2+2_high"} | 瓦数取 power/路数 向上档 |
| cable_group | {SATA:8, SAS:8, NVMe:2} | 前面板线缆每组盘数 |
| gpu_cable_per | {per_model:{"RTX5090":1,"A100":2,"H100":2}} | GPU 供电线单卡根数（型号绑定） |
| bp_type | {mech:["SATA","SAS"]→tri, else→dc} | 背板类型推导 |
| gpu_arch_recommend | {training→switch, inference→pt} | GPU 架构用途推荐（软推荐） |
| switch_constraint | {gpu_required:8} | Switch 必 8 卡（硬约束） |
| switch_extra_parts | {add:["NVSwitch板","NVLink铜缆"]} | Switch 额外料号 |

**约束规则**：盘位匹配（bp specs.bays == config.bays）；底盘件锁定（配置面只读，管理面可改）；电源数量可人工兜底覆盖（留痕）。

## 9. 风险与对策
- **数据迁移丢料号**：迁移脚本 dry-run + 料号数/清单条数校验 + 全库备份。
- **现有整机价格记录功能**：保留，挪管理面下（`price-records` 数据不动）。
- **工作台报价中断**：阶段⑤才切工作台，前 4 阶段工作台仍用旧 ConfigCard，业务不中断。
- **推导规则误判**：所有推导默认值可手改 + 留痕（系统推导/已手动调整），人工兜底开关（电源）。

## 附：已验证的稳定业务规则（来自 4 份报价单 + STEP 文件 + 用户确认）
散热器=CPU数；电源线=电源数；导轨=1；前面板背板=1（规格随机型）；风扇随机型（可选配单列/固定打包底盘）；线缆=按硬盘类型 ÷ group_size（SATA/SAS÷8、NVMe÷2）；GPU 供电线=卡数×单卡线数（5090=1，A100/H100=2）；电源数量=功耗阈值（>4800→4，≤4800→2）；背板类型=含机械盘→三模、纯NVMe→直连；Switch 架构=8 卡全互联+NVSwitch板+NVLink；"直通机型"(GPU拓扑) vs "直连背板"(硬盘层) 独立可交叉。
